1. Resumo Executivo

Este documento apresenta a Analytics Engineering Platform, uma plataforma de dados publicada em github.com/hugoflavio1805/analytics-engineering-platform que materializa, em código produtivo e versionado, o pipeline analítico de um marketplace genérico inspirado em padrões reais (Amazon, MercadoLivre, Shopee, Etsy, eBay).

O princípio orientador é simples e aplicado em todas as camadas: raw stays raw, dashboards stay dumb, every metric has one definition in dbt. As fontes brutas nunca são tocadas; toda derivação é versionada nos modelos dbt; nenhum cálculo de negócio acontece dentro do dashboard.

Headline final: 131 testes dbt verdes em 11,6 segundos · 10 dimensões + 6 fatos + 7 marts analíticos · 7 abas no dashboard com tema dark profissional · tabela de auditoria capturando cada execução do pipeline · integração com Claude API para narrativas com guardrails anti-alucinação.

3.2. dbt-core 1.9 — coração da plataforma

Papel: declarar todas as transformações em SQL versionado e testado, com lineage automático, documentação gerada e separação rigorosa entre staging, intermediate e marts.

Por que dbt e não scripts SQL soltos?

  • Lineage como subproduto: cada ref() declara dependência; o DAG é construído automaticamente.
  • Testes próximos ao modelo: schema.yml ao lado do SQL declara not_null, unique, relationships, accepted_values.
  • Materializações como configuração: alternar entre view, table e incremental é uma linha de YAML.
  • Portabilidade: o mesmo modelo roda em Snowflake, DuckDB, Postgres e Redshift trocando o adapter.
  • Hooks on-run-start e on-run-end: permitem instrumentação da própria pipeline (usado para o audit log).

3.3. DuckDB 1.5 — warehouse analítico embutido

Papel: banco de dados OLAP onde vive toda a camada gold. Executa queries da análise, aplica constraints, suporta transações ACID.

Por que DuckDB e não Postgres?

  • Zero infra: in-process (como SQLite). Sem servidor, sem porta.
  • Performance OLAP: vetorizado em C++, lê Parquet/CSV nativos.
  • Constraints reais: FOREIGN KEY validado, tipos ricos (DECIMAL, TIMESTAMP).
  • Arquivo único: warehouse.duckdb compartilhável, versionável, backup trivial.
  • QUALIFY nativo: deduplicação com window function em uma expressão.

3.4. Snowflake — destino de produção

O profile dbt já tem um target snowflake configurado via env vars. Mesmo SQL que roda local em DuckDB roda em Snowflake sem reescrita. As diferenças entre os dois adapters ficam isoladas em macros/cross_warehouse/qualify_compat.sql.

3.5. Streamlit 1.57 — dashboard interativo

3.7. Anthropic Claude — IA com guardrails

Papel: gerar narrativas executivas para a aba "AI Insights". Modelo padrão é Claude Sonnet 4.6; opcional Opus 4.7 para análises mais profundas.

Disciplina anti-alucinação:

  1. O modelo nunca recebe linhas raw — só recebe payload pré-agregado (top-5 offenders, anomalias, deltas vs baseline).
  2. O system prompt proíbe expressamente: "only cite numbers that appear verbatim in the input".
  3. Cache em disco de 60 min por signature da query — economiza tokens e estabiliza output.
  4. Degradação graciosa: sem ANTHROPIC_API_KEY, a aba mostra o JSON do payload com aviso "AI disabled". O dashboard inteiro continua funcional.

3.8. Docker Compose v5.1 — orquestração local

Dois serviços orquestrados:

  1. dbt-build — one-shot: roda dbt build e popula o volume nomeado aep_warehouse, então sai com exit 0.
  2. dashboard — Streamlit + Plotly + Claude. Depende de dbt-build ter terminado com sucesso (service_completed_successfully).

Volumes nomeados aep_warehouse e aep_ai_cache evitam problemas de permissão de bind-mount no Windows.

4.1. Princípios arquiteturais

PrincípioImplementação
Raw stays raw Os CSVs em data/raw/ são preservados em data/raw_original/. Nenhuma derivação acontece neles. Toda inteligência vive no dbt.
Dashboards stay dumb O Streamlit roda apenas SELECT * FROM mart_*. Zero JOINs, zero agregações, zero lógica de negócio.
One mart per question Cada pergunta de negócio tem seu próprio mart analítico. Quando a definição muda, muda em um lugar só.
Pipeline observability is first-class Cada execução do dbt deixa rastro em main_audit.dbt_runs. A própria plataforma é monitorada como um produto.
AI is a writer, not an analyst Claude recebe payloads pré-agregados e tem instrução explícita de não inventar números.

7. Camada de Staging — Limpeza e Normalização

A camada stg_* tem responsabilidade limitada e bem definida: apenas casts de tipo, normalização de identificadores e flags de DQ. Nenhuma regra de negócio.

7.1. Decisões da camada

  • Materialização: table (não view). Isso torna o warehouse self-contained — o arquivo warehouse.duckdb funciona sem precisar dos CSVs no path original. Crítico para Docker, onde mounts são read-only.
  • Booleanos via try_cast(X as boolean): o read_csv_auto do DuckDB já infere booleans para colunas true/false. case lower(trim(X)) falha porque trim/lower só aceitam VARCHAR.
  • Normalização de IDs: upper(replace(trim(id), '_', '-')) aplicado consistentemente em todas as colunas FK.
  • DQ flags (não filtros): valores problemáticos são marcados (ex: has_negative_price) e não eliminados — preserva o sinal para o singular test detectar.

7.2. Exemplo: stg_orders com parsing dual de datas

{{ config(materialized='table') }}

with src as (
    select * from {{ source('marketplace_raw', 'orders') }}
),
parsed as (
    select
        *,
        coalesce(
            try_cast(order_date as date),
            try_cast(strptime(order_date, '%d/%m/%Y') as date)
        ) as order_date_parsed
    from src
)
select
    order_id, customer_id,
    order_date_parsed                                as order_date,
    lower(status)                                    as status,
    upper(trim(ship_country))                        as ship_country,
    nullif(trim(carrier_id), '')                     as carrier_id,
    nullif(trim(promotion_id), '')                   as promotion_id,
    try_cast(items_count as integer)                 as items_count,
    try_cast(total as decimal(12, 2))                as total
from parsed

8. Camada Intermediária — Regras de Negócio

Materialização ephemeral — não cria tabela física, apenas CTE inline em modelos downstream. Use para regras compartilhadas entre marts.

ModeloRegra de negócio
int_orders_clean Drop pedidos com order_date nulo (parsing falhou). Drop self-cancelled sem payment (phantom orders).
int_returns_classified Calcula days_to_return e is_within_policy (janela de 30 dias). Mapeia reason → reason_category (product_issue / fit_issue / customer_driven / fulfillment_issue).

9. Marts Analíticos — Uma Pergunta por Mart

Cada mart analítico responde a uma pergunta de negócio específica. O dashboard consome diretamente esses marts — não faz JOIN, não agrega, apenas pinta.

9.1. category_region_return_rate

Pergunta: quais (categoria × região) drenam mais GMV via devoluções?

Headline: Fashion + LATAM domina o refund total — alta taxa (27% × 1.10× = 30%) combinada com volume razoável. Books/APAC é o quadrante mais limpo.

9.2. gmv_by_promotion

Pergunta: quais campanhas têm ROI positivo após chargebacks/devoluções?

Headline: 31 campanhas no dataset; uma única campanha agressiva responde por mais GMV que toda a soma das free-shipping, mas com 18% de chargeback (vs 4% baseline). Net contribution é a métrica certa, não GMV bruto.

9.3. sla_by_carrier

Pergunta: qual carrier mais erra SLA, e como isso correlaciona com devoluções por atraso?

Headline: Correios (BR) tem reliability_score 0.72 e taxa de SLA miss alta. Substituir por Sedex (0.83) em rotas BR↔BR é a alavanca de retenção mais barata em fulfillment.

9.4. customer_cohort_retention

Pergunta: como cohorts de aquisição retêm ao longo do tempo?

Headline: matriz de 1.042 rows (cohort_month × months_since_acquisition). Renderizada como heatmap dos últimos 12 cohorts no dashboard.

9.5. pareto_sellers

Pergunta: qual % de sellers domina qual % de GMV?

Headline: 992 sellers ranqueados; bucket A (top sellers até 50% do GMV cumulativo) tipicamente é ~5% dos sellers. Validação clássica da regra 80/20.

9.6. growth_metrics_monthly

Pergunta: como o negócio cresce MoM/YoY?

Headline: 52 meses de série temporal (2022-2026). Inclui mom_growth_pct, yoy_growth_pct, annualized_run_rate.

9.7. customer_ltv_by_segment

Pergunta: qual o LTV líquido (lifetime_gmv menos refunds) por segmento?

Grão: 1 row por (região × is_prime × tenure_bucket). 32 segmentos no total.

10. Pipeline Observability (Audit Infrastructure)

Esta é a camada que diferencia uma plataforma analítica madura de um conjunto de notebooks. Cada execução do dbt deixa rastro auditável.

10.1. Como funciona

Duas macros customizadas em dbt/macros/audit/persist_run_results.sql:

  • ensure_audit_table() — roda em on-run-start. Cria a tabela main_audit.dbt_runs se não existir. Garante que mesmo o primeiro dbt build do projeto já tenha onde gravar.
  • persist_run_results() — roda em on-run-end. Itera sobre os results da invocação e insere uma linha por nó executado.

10.2. Schema de main_audit.dbt_runs

10.3. Mart de saúde da pipeline

O mart pipeline_health (view) agrupa o audit log por run_id e calcula:

  • pass_count, warn_count, error_count, skipped_count
  • total_duration_seconds
  • overall_status: green · warning · failed

10.4. O que aparece no dashboard

A aba Pipeline Health renderiza:

  • Hero card com a última execução: status colorido (verde/laranja/vermelho), PASS/WARN/ERROR como métricas, duração, git_sha
  • Timeline dos últimos 25 runs como scatter horizontal colorido por status
  • Stacked area de PASS/WARN/FAIL ao longo do tempo (test outcome history)
  • Slowest models (top 10 por duração média)
  • Last run breakdown: tabela completa node-by-node do último run
Por que isso importa: em uma plataforma de produção, debugar pipeline lento ou test flaky exige histórico granular. main_audit.dbt_runs é exatamente como Datadog APM, Snowflake QUERY_HISTORY ou Airbyte logs funcionam — observabilidade da pipeline como cidadã de primeira classe.

Lift de chargeback de aggressive vs baseline = +14,46 pontos percentuais. Em termos relativos, é um aumento de 367% na taxa de fraude.

Recomendação operacional baseada nos 3 painéis:
  1. Aprovação obrigatória de Finance para qualquer campanha com is_aggressive=true rodando em janelas de Black Friday / Cyber Monday.
  2. Default sazonal Q4 deve ser free-shipping, não percentage discount.
  3. Bloquear method=crypto em checkout durante janelas de aggressive promos (combo dispara chargeback).
  4. Monitorar AOV médio por bucket em tempo real — picos súbitos em aggressive são sinal de fraude organizada.

13. AI Insights com Anthropic Claude

A aba "AI Insights" usa a Claude API (modelo Sonnet 4.6 por padrão) para gerar narrativas executivas em estilo OP1 da Amazon. Esta é a peça mais sofisticada da plataforma do ponto de vista de safety.

13.1. Disciplina anti-alucinação

Quatro guardrails complementares:

  1. Modelo nunca vê linhas raw. O ai_insights.py tem funções payload_for_returns(), payload_for_promotions(), etc. que extraem apenas top-N e agregados.
  2. System prompt explícito: "Only cite numbers that appear verbatim in the input data. If a number isn't in the input, do not invent it."
  3. Estilo OP1: 4 parágrafos máximo, "so-what" primeiro, watch items no final, recomendação como hipótese testável.
  4. Cache em disco de 60 minutos por signature (hash do payload). Evita gastar tokens em renders repetidos do mesmo dado.

14. Containerização e Execução com Docker Compose

14.1. Arquitetura do compose

services:
  dbt-build:
    image: dataengineer-dbt-build (built from docker/Dockerfile.dbt)
    volumes:
      - ./dbt:/workspace/dbt
      - ./cases:/workspace/cases:ro
      - aep_warehouse:/warehouse
    environment:
      DBT_DUCKDB_PATH:     /warehouse/warehouse.duckdb
      MARKETPLACE_RAW_DIR: /workspace/cases/marketplace/data/raw
    command: build --profiles-dir . --target duckdb

  dashboard:
    image: dataengineer-dashboard (built from app/Dockerfile)
    depends_on:
      dbt-build: { condition: service_completed_successfully }
    ports: ["8501:8501"]
    volumes:
      - aep_warehouse:/warehouse:ro
      - aep_ai_cache:/app/.ai_cache
    env_file:
      - path: ./.env
        required: false
    environment:
      DUCKDB_PATH:      /warehouse/warehouse.duckdb
      ANTHROPIC_MODEL:  ${ANTHROPIC_MODEL:-claude-sonnet-4-6}

volumes:
  aep_warehouse:
  aep_ai_cache:

14.2. Decisões de containerização

  • Volumes nomeados (não bind mounts) para o warehouse e cache. Resolve permissões no Windows.
  • Compose v2 dependency condition (service_completed_successfully) garante que o dashboard só sobe depois do dbt-build terminar com exit 0.
  • Source CSVs montados read-only — defesa contra escrita acidental.
  • Mesma stack roda em desenvolvimento e em CI — diferença é apenas o target dbt (duckdb em dev, snowflake em prod).

14.3. Comando único de execução

cd /c/DataEngineer && \
docker compose -f docker-compose.dashboard.yml down && \
docker compose -f docker-compose.dashboard.yml up -d

Abre http://localhost:8501. Stack ~12s para build dbt + 4s para subir dashboard.

16. CI/CD e GitHub Actions

Workflow em .github/workflows/ci.yml roda em todo PR e em todo push para main:

  1. Setup Python 3.11 + cache de pip
  2. Install requirements.txt
  3. SQL lint com sqlfluff (não-bloqueante hoje, pronto para virar bloqueante)
  4. dbt deps + dbt parse — captura erros Jinja antes da execução
  5. dbt build --target ci — full build + tests contra DuckDB efêmero
  6. Upload artifacts do target/ para depuração

No merge para main, job adicional roda dbt docs generate e publica em GitHub Pages.

Custo de CI: ~3 minutos por PR no tier gratuito (público). Suficiente para 600+ PRs/mês dentro dos 2.000 minutos gratuitos.

17.2. Como escala para 10×

  • Compute: Snowflake XS aguenta sem trocar de tamanho até ~5M linhas/dia; depois sobe para S.
  • Incremental models em fct_orders garantem que o custo cresce com delta, não com tamanho da tabela.
  • BI: Streamlit, Tableau e Metabase escalam linearmente com cliques de usuário, não com tamanho do warehouse.
  • AI cache: hit rate alto reduz custo Claude para ~zero em browsing sessions concentradas.

19. Limitações Conhecidas e Roadmap Futuro

19.1. Limitações

  • Volume sintético — não substitui dados reais para análises causais rigorosas.
  • fct_orders incremental requer --full-refresh ao regenerar dados (regra geral de incrementals).
  • Aba AI Insights depende de créditos Anthropic — sem eles a aba degrada para JSON.
  • Tableau Public ainda é placeholder — link aparece em cases/marketplace/dashboards/ mas não tem dashboard publicado.
  • Geographic granularity para choropleth é por país — falta breakdown por estado/região.
  • Audit log não tem retenção configurada — em produção precisa delete from main_audit.dbt_runs where invocation_at < current_date - 90 day.

19.2. Roadmap próximos passos

  1. Reverse ETL — exportar marts para CRM (HubSpot/Salesforce) via Hightouch ou Census.
  2. Great Expectations ao lado dos dbt tests, para monitorar distribuição (não só constraints).
  3. dbt Semantic Layer / MetricFlow — definição única de GMV, return-rate, churn-rate consumida por qualquer BI.
  4. Anomaly detection em volume de pedidos por canal — silêncio súbito é sinal por si só.
  5. Cohort retention curves materializadas e atualizadas diariamente.
  6. Tableau Public dashboard publicado de fato (executive view paralela ao Streamlit).
  7. Snowflake target testado de ponta a ponta — hoje só foi validado em DuckDB.
  8. RBAC no Streamlit — diferentes níveis de visualização por papel (Finance vê GMV, Ops vê SLA, etc.).
  9. Multi-tenant audit — separar runs por tenant em deployments com mais de 1 cliente.

20. Glossário Técnico

GMV
Gross Merchandise Value — valor total transacionado, antes de descontos e devoluções.
AOV
Average Order Value — GMV ÷ número de pedidos.
AUO
Average Units per Order — total de unidades vendidas ÷ pedidos.
LTV
Lifetime Value — receita líquida estimada de um cliente ao longo de toda sua vida na plataforma.
Chargeback
Disputa de cartão de crédito iniciada pelo titular junto ao banco. Indicador clássico de fraude.
Cohort
Grupo de clientes adquiridos em um mesmo período (ex: cohort de Janeiro/2024). Usado para análise de retenção.
Pareto / 80-20
Princípio empírico de que ~80% do efeito vem de ~20% das causas. Em marketplaces, top 5% dos sellers tipicamente respondem por 50% do GMV.
MoM / YoY
Month-over-Month / Year-over-Year — taxas de crescimento.
Run rate
Receita anualizada projetada a partir do mês corrente (mês × 12).
Staging (dbt)
Camada 1:1 com a fonte raw. Apenas casts, normalização. Sem regra de negócio.
Mart (dbt)
Tabela final consumida por BI. Pode ser core (dims/facts conformados) ou analytics (uma por pergunta).
Incremental model
Materialização dbt que processa apenas linhas novas a cada execução. Reduz custo de warehouse.
Source freshness
Verificação de que a fonte de dados foi atualizada dentro de uma janela esperada (warn/error).
Singular test
Teste dbt escrito como SELECT — falha se retornar qualquer linha. Usado para invariantes complexas.
QUALIFY
Cláusula SQL (Snowflake/DuckDB) que filtra resultados de window functions sem CTE intermediário.
Lineage
Grafo de dependências entre modelos. dbt deriva automaticamente via ref().
Choropleth
Mapa colorido por valor numérico de cada região geográfica.
Dual-axis chart
Gráfico com dois eixos Y (escalas diferentes) — comum para sobrepor métricas relacionadas mas em unidades diferentes.
BOGO
"Buy One Get One" — promoção em que o cliente leva 2 unidades pelo preço de 1.
OP1 / OP2
Operating Plan reviews na Amazon — documentos narrativos curtos com foco em "so-what" e ação.
Working Backwards
Princípio Amazon de partir da experiência do cliente final e desenhar para trás até a infraestrutura.
